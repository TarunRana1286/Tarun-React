OOP Fundamentals
1. What is Object-Oriented Programming (OOP)?
OOP is a programming paradigm that organizes software design around objects rather than functions and logic. Objects contain data (properties/fields) and behavior (methods).
The four main pillars of OOP are:

Encapsulation
Abstraction
Inheritance
Polymorphism

In C#, everything revolves around classes and objects, promoting reusability, maintainability, and modeling real-world entities.
2. What is the difference between procedural and Object-Oriented programming?

Procedural: Code is organized around functions/procedures. Data and behavior are separate (e.g., C, older VB). Focuses on "what to do" step-by-step. Harder to maintain for large systems.
Object-Oriented: Code is organized around objects that combine data and behavior. Focuses on "who is doing what." Better for complex, evolving systems due to modularity and reuse.

Example: In procedural style, you might have separate functions to calculate area. In OOP (C#), a Shape class owns the data and the CalculateArea() method.
3. What is encapsulation?
Encapsulation is the bundling of data (fields) and methods that operate on that data into a single unit (class), while restricting direct access to some components (data hiding).
It protects internal state using access modifiers (private, protected, etc.) and exposes controlled access via properties or methods.
C#public class BankAccount
{
    private decimal _balance; // hidden field

    public decimal Balance 
    { 
        get => _balance; 
        private set => _balance = value; // controlled setter
    }

    public void Deposit(decimal amount)
    {
        if (amount > 0) _balance += amount;
    }
}
4. What is polymorphism? Explain overriding and overloading.
Polymorphism means "many forms" — the ability to treat objects of different types as instances of the same base type.

Overloading (Compile-time / Static): Same method name, different parameters (or types).
Overriding (Runtime / Dynamic): Child class redefines a virtual/base method.

C#public class Calculator
{
    public int Add(int a, int b) => a + b;           // Overloading
    public double Add(double a, double b) => a + b;  // Overloading
}

public class Animal
{
    public virtual void MakeSound() => Console.WriteLine("Animal sound");
}

public class Dog : Animal
{
    public override void MakeSound() => Console.WriteLine("Bark"); // Overriding
}
5. What is inheritance? Name some types of inheritance.
Inheritance allows a class (derived/child) to acquire properties and methods from another class (base/parent), promoting code reuse and hierarchy.
In C# (single inheritance for classes):

Single — One base class.
Multi-level — A → B → C.
Hierarchical — Multiple classes inherit from one base.

C# does not support multiple inheritance of classes (to avoid diamond problem), but supports it via interfaces. Use :  for inheritance.
6. What is an abstraction? Name some abstraction techniques.
Abstraction hides complex implementation details and shows only essential features. It reduces complexity and allows focusing on "what" instead of "how."
Techniques in C#:

Abstract classes (with abstract methods).
Interfaces.
Properties and public methods that hide internals.

C#public abstract class Shape
{
    public abstract double CalculateArea(); // Abstract method
}

public class Circle : Shape
{
    public double Radius { get; set; }
    public override double CalculateArea() => Math.PI * Radius * Radius;
}
7. What is a class in OOP?
A class is a blueprint or template that defines the structure (properties, fields) and behavior (methods) of objects. It acts as a user-defined data type.
In C#: Classes support constructors, destructors (finalizers), inheritance, etc.
8. What is an object in OOP?
An object is an instance of a class — a real entity in memory that has state (data) and behavior (methods).
Example: Circle myCircle = new Circle { Radius = 5 };
9. How do access specifiers work and what are they typically?
Access specifiers (modifiers) control visibility and accessibility:

private — Only within the same class.
protected — Within the class and derived classes.
internal — Within the same assembly.
protected internal — Derived classes or same assembly.
public — Everywhere.
private protected — Same assembly + derived (C# 7.2+).

They enforce encapsulation.
10. Name some ways to overload a method.

Different number of parameters.
Different types of parameters.
Different order of parameters (when types differ).

Constructors can also be overloaded. The compiler resolves at compile time based on signature.
11. What is cohesion in OOP?
Cohesion measures how closely related the responsibilities of a class/module are. High cohesion is desirable — a class should have a single, well-defined purpose (aligns with Single Responsibility Principle).
12. What is coupling in OOP?
Coupling measures dependency between classes/modules. Low coupling is desirable — changes in one class should not heavily impact others. Achieved via interfaces, dependency injection, and composition.
Class Relationships and Design
13. What is a constructor and how is it used?
A constructor is a special method called automatically when an object is created (new). It initializes fields/properties.
Types in C#: Default, Parameterized, Copy, Static, Private.
C#public class Person
{
    public string Name { get; }
    public Person(string name) => Name = name; // Parameterized
}
14. Describe the concept of destructor or finalizer in OOP.
In C#, there is no traditional destructor like in C++. Instead, we use finalizers (~ClassName()) for cleanup (rarely used). Better alternatives: IDisposable + Dispose() pattern for unmanaged resources.
The garbage collector calls the finalizer automatically.
15. Compare inheritance vs. mixin vs. composition.

Inheritance: "Is-a" relationship (tight coupling, code reuse via hierarchy).
Mixin: Reusable behavior added via interfaces or extension methods (C# has no true mixins like some languages; use default interface methods in C# 8+).
Composition: "Has-a" relationship (preferred — more flexible, "favor composition over inheritance"). Objects contain other objects.

16. Explain the concept of an interface and how it differs from an abstract class.

Interface: Defines a contract (methods, properties, events) without implementation (until C# 8 default methods). Multiple inheritance supported. No fields, no constructors.
Abstract class: Can have implementation, fields, constructors, and protected members. Single inheritance only.

Use interfaces for behavior contracts; abstract classes for shared implementation + partial abstraction.
17. Can a class have multiple parents in a single-inheritance system?
No, C# supports single class inheritance only. However, a class can implement multiple interfaces, achieving a form of multiple inheritance.
18. How would you design a class to prevent it from being subclassed?
Use the sealed keyword:
C#public sealed class UtilityHelper { ... }
// Cannot be inherited from
19. Explain the ‘is-a’ vs ‘has-a’ relationship in OOP.

Is-a: Inheritance (e.g., Dog : Animal).
Has-a: Composition/Aggregation (e.g., Car has an Engine — private readonly Engine _engine;).

20. Explain how aggregation relationship is represented in OOP.
Aggregation is a "has-a" relationship where the part can exist independently of the whole (weaker than composition). In C#, it's typically a reference or collection:
C#public class Department
{
    public List<Employee> Employees { get; } = new(); // Aggregation
}
21. What is method overriding, and what rules apply to it?
Overriding allows a derived class to provide a specific implementation of a virtual or abstract method from the base class. Use virtual/abstract in base and override in derived.
Rules: Same signature, cannot override non-virtual/static methods, accessibility cannot be reduced.
22. Describe the use of static methods and when they are appropriate.
static methods belong to the class, not instances. No this reference. Useful for utility/helper methods, factories, or when state is not needed (e.g., Math.Sqrt(), extension methods). Avoid if you need polymorphism or instance state.
Advanced OOP Concepts
23. What is multiple inheritance and what are some of its disadvantages?
Multiple inheritance = one class inheriting from multiple base classes. C# does not support it for classes (only interfaces) to avoid complexity.
Disadvantages: Diamond problem, ambiguity, tight coupling, maintenance issues.
24. Can you explain the ‘diamond problem’ in multiple inheritance?
When two base classes inherit from the same grandparent, and a derived class inherits from both — ambiguity arises on which grandparent method/property to use. C# avoids this by using interfaces + explicit implementation if needed.
25. How does OOP languages support polymorphism under the hood?

Compile-time (overloading): Resolved by compiler via method signature.
Runtime (overriding): Uses virtual method table (vtable) — each object has a pointer to the correct method implementation.

In C#, virtual + override enables dynamic dispatch.
26. What are generics, and how can they be useful in OOP?
Generics allow type-safe reusable code without boxing/unboxing or casting.
C#public class Repository<T> where T : class
{
    public void Add(T item) { ... }
}
Benefits: Type safety, performance, code reuse (e.g., List<T>, Dictionary<TKey, TValue>).
27. Explain the concept of object composition and its benefits.
Composition: A class contains instances of other classes as members ("has-a").
Benefits: Greater flexibility than inheritance, easier to change behavior at runtime, follows "favor composition over inheritance," better encapsulation.
28. What is Liskov Substitution Principle (LSP)? Provide some examples of violation and adherence.
LSP (part of SOLID): Objects of a derived class should be substitutable for objects of the base class without altering the correctness of the program. Subtypes must honor the base class contract.
Violation example (classic Rectangle-Square):
C#public class Rectangle
{
    public virtual int Width { get; set; }
    public virtual int Height { get; set; }
    public int GetArea() => Width * Height;
}

public class Square : Rectangle
{
    public override int Width { set => base.Width = base.Height = value; }
    public override int Height { set => base.Width = base.Height = value; }
}
// Substituting Square breaks expectations in code expecting independent Width/Height.
Adherence: Use proper hierarchy or interfaces where behavior remains consistent.
29. What is the dependency inversion principle?
High-level modules should not depend on low-level modules. Both should depend on abstractions (interfaces/abstract classes). Abstractions should not depend on details; details should depend on abstractions.
Achieved via Dependency Injection (DI) in .NET (e.g., constructor injection).
30. How can the open/closed principle guide object-oriented design?
Open/Closed Principle (OCP): Software entities (classes, modules, functions) should be open for extension but closed for modification.
Implement by using interfaces, abstract classes, and polymorphism/Strategy pattern instead of modifying existing code.
31. Describe how the Interface Segregation Principle affects system design.
Interface Segregation Principle (ISP): Clients should not be forced to depend on interfaces they do not use. Prefer many small, specific interfaces over one large "fat" interface.
This reduces coupling and prevents implementing unused methods.
32. What is a mixin, and how does it differ from traditional inheritance?
A mixin adds reusable behavior to classes without full inheritance hierarchy. C# doesn't have native mixins, but you can simulate with:

Default interface methods (C# 8+).
Extension methods.
Composition.

Unlike inheritance, mixins avoid deep hierarchies and diamond issues.
Practical Use and Patterns
33. How would you refactor a class that has too many responsibilities?
Apply Single Responsibility Principle (SRP): Split the class into smaller classes, each with one responsibility. Use composition or delegation. Extract methods into services or use patterns like Strategy/Factory.
34. Describe a singleton pattern and discuss its pros and cons.
Singleton ensures a class has only one instance and provides global access.
C#public sealed class Logger
{
    private static readonly Lazy<Logger> _instance = new(() => new Logger());
    public static Logger Instance => _instance.Value;
    private Logger() { }
}
Pros: Controlled access, lazy initialization.
Cons: Global state (hard to test), violates SRP sometimes, threading issues if not careful (use Lazy<T> or lock).

35–42 (Design Patterns summary in C# context):

Factory Method: Creates objects without specifying exact class (e.g., CreateShape() returning different IShape implementations).
Builder: Step-by-step construction of complex objects (fluent API for objects with many optional params).
Prototype: Clone existing objects (use ICloneable or copy constructor).
Adapter: Convert interface of a class into another expected by client.
Decorator: Add behavior dynamically (wrapper classes, e.g., logging decorator).
Observer: One-to-many dependency (e.g., events/delegates in C#, or IObserver<T>).
Command: Encapsulate request as object (undo/redo support).
Strategy: Define family of algorithms, make them interchangeable (e.g., different sorting strategies).

Example Strategy Pattern:
C#public interface ISortStrategy { void Sort(List<int> list); }

public class QuickSort : ISortStrategy { public void Sort(List<int> list) { ... } }
OOP Best Practices
43. What are some common OOP design anti-patterns?

God Class (too many responsibilities).
Anemic Domain Model (data-only classes with logic elsewhere).
Spaghetti Code.
Magic Strings/Numbers.
Tight Coupling.
Overuse of inheritance (deep hierarchies).

44. How do you ensure that your objects are properly encapsulated?

Make fields private.
Use properties with getters/setters (with validation).
Expose behavior via methods, not direct data.
Avoid public fields.

45. Name some techniques for reducing coupling between classes.

Use interfaces/abstractions.
Dependency Injection (constructor, property, method).
Event-driven architecture.
Composition over inheritance.
Mediator or Facade patterns.

46. How does immutability help in object-oriented design, and how can it be implemented?
Immutability (objects cannot change after creation) improves thread-safety, predictability, and simplifies reasoning about code.
In C#: Use readonly fields, init-only properties ({ get; init; }), records (record Person(string Name);), or private setters + constructor-only initialization.
47. What tools or techniques would you use to document an object-oriented design?

UML diagrams (class, sequence).
XML comments (///).
Tools: Visual Studio, PlantUML, Draw.io, Swagger for APIs.
Code-level: Meaningful names, interfaces as contracts.

48. How do you address circular dependencies in an OOP system?

Refactor to introduce interfaces or a mediator.
Use events or message queues.
Dependency Injection with factories.
Split assemblies/modules.

49. Explain how to apply unit testing to object-oriented code.

Test small units (methods/classes) in isolation using mocks (Moq, NSubstitute).
Inject dependencies via interfaces.
Follow AAA (Arrange-Act-Assert).
Test behavior, not internals.
Use xUnit/NUnit + FluentAssertions.

50. What strategies can be used to safely refactor legacy object-oriented code?

Add tests first (characterization tests).
Small, incremental changes (boy scout rule).
Use patterns like Strangler Fig (gradually replace old code).
Introduce interfaces for legacy classes.
Apply SOLID gradually.

51. How can the principles of OOP help in achieving a modular and maintainable codebase?
OOP promotes:

Modularity via classes/interfaces.
Reusability via inheritance/composition.
Maintainability via encapsulation + SOLID.
Extensibility via polymorphism and OCP.

Result: Easier testing, scaling, and team collaboration.
52. How do you balance the use of OOP principles with performance considerations in a system design?

Use composition and interfaces wisely (virtual calls have slight overhead).
Prefer structs/value types for small, immutable data (stack allocation).
Avoid deep inheritance or excessive polymorphism in hot paths.
Profile with tools (dotnet-trace, Visual Studio Profiler).
Use records, readonly structs, and Span<T> for performance-critical code.
Apply OOP where maintainability matters most; optimize bottlenecks separately.
